# 新增表与修改表设计计划

本文把新增数据表、修改数据表两条主线拆成可实现的 GUI 计划，作为后续实现表设计器的工作基线。

## 最高边界

- 新增表和修改表结构只走结构化设计器，不要求用户编写或审阅底层执行文本。
- 人类界面只展示表单、字段网格、索引网格、约束编辑器、选项、校验、变更计划、风险确认和执行结果。
- 执行文本只允许作为内部驱动实现细节，不出现在设计器页签、弹窗、动作条、复制动作、失败建议或默认表详情中。
- 所有变更进入 `Schema Change Plan`，保存前可见、可回退、可确认。
- 驱动不支持的能力必须显示原因，不伪装成可执行。
- 成功后刷新元数据，保持用户在刚创建或修改的表上。

## 当前落地边界

已落地：

- app 层已有 `SchemaChangePlan`，由新建表和修改表共用。
- 新建表设计器显示对象、目标连接、结构摘要、驱动能力、执行路径和逐条计划。
- 修改表设计器显示当前操作、风险、阻塞原因、动作数量和元数据刷新步骤。
- 阻塞计划不会进入执行路径。
- 修改表高风险计划会先进入确认弹窗，确认后才执行同一份内部计划。
- 修改表支持回退选中变更和回退全部变更；当前基于单一待执行操作模型恢复字段、索引或约束草稿。
- 执行成功后会刷新 schema 元数据。

尚未落地：

- 持久化 schema 执行日志。
- 修改数据库完整设计器。
- MongoDB 独立 collection designer。

## 新增表主线

入口：

- 数据库节点右键 -> `新建数据表`
- schema 节点右键 -> `新建数据表`
- Tables group 右键 -> `New Table`
- 顶栏 `New` -> `Table`

设计器区域：

| 区域 | 内容 |
|---|---|
| 头部（Header） | 目标连接、database、schema、驱动、测试延迟 |
| 动作条（Action Bar） | 保存草稿、回退、创建表、更多 |
| 名称行（Name Row） | 表名、注释、目标 schema/database |
| 字段（Fields） | 字段网格 |
| 索引（Indexes） | 索引网格 |
| 约束（Constraints） | 主键、外键、唯一、检查约束 |
| 选项（Options） | engine、charset、collation、tablespace、partition、comment |
| 变更计划 | 对象、字段、索引、约束、驱动能力、风险和执行顺序 |
| 校验/结果 | 校验消息、执行结果、元数据刷新结果 |

字段网格：

| 列 | 行为 |
|---|---|
| 状态/PK | 标记主键和草稿状态 |
| Name | 必填，禁止重复 |
| Type | 驱动感知下拉 |
| Length | 按类型启用或禁用 |
| Precision | 按类型启用或禁用 |
| Scale | 按类型启用或禁用 |
| Not Null | 布尔控件 |
| Auto Inc / Identity | 按驱动能力启用 |
| Default | 支持表达式或字面量 |
| Charset/Collation | MySQL-family 字符字段启用 |
| Comment | 字段备注 |

执行步骤：

1. 用户进入新建表设计器，系统带入目标上下文。
2. 系统生成默认字段草稿，例如 `id`，但允许删除。
3. 用户填写表名、字段、索引、约束和选项。
4. 系统即时校验字段重复、主键冲突、类型参数是否合法。
5. 变更计划面板展示对象路径、结构摘要、驱动能力、风险和执行顺序。
6. 用户点击 Create Table。
7. 系统按内部驱动计划执行创建动作。
8. 系统刷新目标 database/schema 的对象树。
9. 系统选中新表并打开表详情或表设计器。

## 修改表主线

入口：

- 表节点右键 -> `设计数据表`
- 表详情页动作条 -> `Design`
- 数据浏览页顶部 -> `Structure`

设计器区域：

| 区域 | 内容 |
|---|---|
| 头部（Header） | 表路径、行数、驱动、元数据刷新耗时 |
| 动作条（Action Bar） | 保存草稿、回退选中、回退全部、应用变更 |
| 字段（Fields） | 可编辑字段网格 |
| 索引（Indexes） | 索引列表和索引字段 |
| 约束（Constraints） | 主键、外键、唯一、检查约束 |
| 选项（Options） | 表选项、分区、存储参数 |
| 变更计划（Change Plan） | 将执行的结构动作、风险和阻塞原因 |
| 历史（History） | 已执行的结构变更记录 |

变更标记：

| 标记 | 含义 | 是否需要确认 |
|---|---|---|
| `=` | 未变化 | 否 |
| `+` | 新增 | 否 |
| `~` | 修改 | 视风险而定 |
| `-` | 删除 | 是 |
| `!` | 高风险或驱动不完全支持 | 是 |

高风险项：

- 删除字段。
- 修改字段类型。
- 缩短字段长度。
- 将 nullable 改为 not null。
- 删除索引。
- 删除约束。
- 移动字段位置。
- 修改字符集/排序规则。
- 可能重写整表的操作。

执行步骤：

1. 用户打开表设计器。
2. 系统加载实时字段、索引、约束和表选项。
3. 用户编辑字段、索引、约束或表选项。
4. 每个编辑保留原值、当前值、动作类型和风险级别。
5. 变更计划按驱动执行顺序生成。
6. 用户可以回退选中变更或回退全部变更。
7. 用户点击应用变更。
8. 系统对高风险项显示确认，确认内容包含完整表路径、风险项和结构动作数量。
9. 系统执行内部驱动计划。
10. 系统刷新表结构和对象树。
11. 系统记录执行日志并保留设计器上下文。

## 驱动差异

### MySQL 系列

- 支持 engine、charset、collation、comment。
- 支持字段位置 `FIRST` / `AFTER`。
- TiDB 需要额外标注分布式能力和不支持的 MySQL 语法。
- MariaDB 需要保留 sequence 等扩展对象的后续入口。

### PostgreSQL 系列

- schema 是必选上下文。
- 类型、default、identity、generated column、sequence 需要驱动感知。
- 表选项包含 tablespace、owner、storage parameters。
- 某些类型变更需要额外表达式时，必须用结构化字段采集，不能静默生成。

### SQL Server 系列

- schema 是必选上下文。
- identity、computed column、clustered/nonclustered index 是重要字段。
- 加密、兼容级别、legacy mode 影响可用能力。

### MongoDB

- 不进入关系型表设计器。
- collection 创建、validator、indexes、document sample 应单独映射为 collection designer。

## 支线 UI

支线 UI 不抢主线，但必须可发现：

- 变更计划面板：查看变更顺序、风险、驱动支持状态。
- 执行日志面板：查看执行结果、耗时、错误阶段。
- 驱动能力检查器：说明当前驱动支持/不支持哪些表设计能力。
- 类型选择器：按驱动分组的数据类型选择器。
- 索引字段选择器：选择索引字段、顺序、排序方向。
- 约束编辑器：配置外键、唯一、检查约束。
- 风险确认弹窗：高风险操作确认。

## 验收要点

- 新增表不需要用户打开查询编辑器。
- 修改表不把用户带离当前对象上下文。
- 新增表和修改表结构界面不展示、复制或要求审阅底层执行文本。
- 变更计划清楚显示每一项变更。
- 高风险操作必须确认。
- 执行成功后对象树刷新并选中目标表。
- 执行失败显示阶段、错误码/错误类、可读消息、是否部分执行。
