# 设计 Unix 风格工作台 UX

## 为什么

Akt 需要在继续实现更多数据库功能前，先形成一套逻辑自洽、可实现、可验收的 UX 原型计划。该计划必须以 MySQL、PostgreSQL、MongoDB、SQL Server 官方模型为约束，提炼最大公约数，而不是把所有数据库硬塞进单一 SQL 表结构视图。

当前连接管理、对象树、查询控制台和外观设置已有局部实现，但还缺少一个统一产品层设计来约束：

- 连接参数如何按数据库差异自然呈现。
- 数据库对象层级如何在 SQL 与文档数据库之间保持一致。
- 用户动作如何从对象定位自然过渡到右侧工作台操作。
- 真实连接延迟如何进入连接测试、状态栏和连接列表。
- 字体如何从本地系统选择，而不是打包进应用增加体积和内存。
- 不同 theme 如何拥有不同字体、密度和可读性策略。

## 变更内容

- 增加 `unix-workbench-ux` 能力，作为后续连接、对象树、查询、设置、字体和延迟实现的 UX 设计基线。
- 定义最大公约数对象模型：`Connection -> Namespace -> Container -> Object Group -> Object`。
- 定义四类数据库的映射方式：MySQL、PostgreSQL、MongoDB、SQL Server。
- 定义自然操作流：左侧定位对象，右侧执行上下文动作，危险操作强确认。
- 定义真实延迟指标：连接延迟、往返延迟、元数据刷新延迟。
- 定义系统字体策略：macOS、Windows、Linux 从本地系统选择 UI 字体和等宽字体，不打包字体。
- 定义 theme 与字体/密度的关系。

## 非目标

- 本变更不实现数据库驱动。
- 本变更不改现有 Rust UI 代码。
- 本变更不决定最终商业功能边界。
- 本变更不要求一次性完成所有数据库对象操作。

## 影响

- 受影响 specs：`unix-workbench-ux`
- 后续会影响代码：`connection`、`drivers`、`metadata`、`query`、`theme`、`settings`、`ui`
- 后续会影响 UI：连接管理、对象树、查询控制台、状态栏、设置页
- 后续会影响发布包：移除内置字体资产，改为系统字体发现与 fallback
