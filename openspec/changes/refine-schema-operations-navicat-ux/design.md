# 设计

## 全量 OpenSpec 对齐结论

已逐项读取现有 OpenSpec：

- `desktop-app-shell`：定义 Rust/iced 主壳、顶部栏、侧栏、中央工作区、状态栏；粒度偏基础。
- `connection-manager`：定义连接模型、驱动默认值、连接表单和 SQLite 测试；连接参数完整度还需要后续按驱动扩展。
- `shell-stability-and-i18n`：固定 1280x800 桌面窗口、原生窗口装饰、有界布局、稳定配置路径；这是本变更的布局硬约束。
- `appearance-settings`：设置页已有 theme、density、typography、keybindings 入口；字体发现和 theme 字体差异由 `unix-workbench-ux` 后续承接。
- `object-workbench`：定义对象树作为左侧定位器、创建数据库直接执行、危险操作确认、数据库/数据表动作；本变更细化它的 schema 操作体验。
- `query-console`：定义 SQL 控制台、SQLite 查询、schema browser 和响应式执行；本变更不得把 schema 操作退回查询编辑器。
- `unix-workbench-ux`：定义最大公约数对象模型、真实延迟、系统字体发现和 theme 策略；本变更继承其对象路径和状态栏语义。
- `add-official-site-and-release-pipeline`：只影响官网和发布管线，不参与桌面 UX 约束。

颗粒度修正：

- 现有 `object-workbench` 已规定做什么，但没有规定 schema 设计器每个区域的尺寸、按钮位置、tab 顺序、变更计划和确认节奏。
- 现有 `query-console` 已规定查询编辑器不是 schema 操作主入口，本变更进一步固定：新增表和修改表结构不得把底层执行文本作为人的审阅面板。
- 现有 `shell-stability-and-i18n` 固定窗口尺寸，本变更所有像素规格以 1280x800 内容窗口为基准。
- 后续实现不得新增第二套 schema 操作入口；本变更定义的入口和动作条为唯一主路径。

## Navicat 对齐基准

本设计参考 Navicat Premium 的官方产品描述：其工作方式强调多数据库连接、Object Designer、Data Editor、Structure Synchronization、Native Design 和 Dark Mode。这里不复制 Navicat 的品牌资产、图标或专有视觉细节；对齐的是专业数据库客户端的一比一操作节奏：对象浏览器进入、结构化设计器编辑、数据/结构分离、变更可审阅、执行反馈明确。

参考来源：

- https://www.navicat.com/en/products/navicat-premium

## 总体交互原则

1. 对象树是入口，不是唯一工作区。
2. 新建操作可以是弹窗或右侧设计器，但必须保留当前数据库上下文。
3. 修改操作优先使用右侧工作台内嵌设计器，避免用户在多个弹窗之间迷路。
4. 表单是主路径，变更计划、校验、风险和结果是人的审阅对象。
5. 保存前必须能看到“将执行什么变更”。
6. 执行后必须刷新对象树，并保持用户在刚操作的对象上。
7. 支持的操作直接执行；不支持的操作明确说明原因，不伪装成功。

## 流程一：新建数据库

入口：

- 连接节点右键 -> `新建数据库`
- 顶栏 `New` -> `Database`
- 空工作台中的 `Create Database`

像素级布局：

```text
工作台 240,52,1040,718
标题栏 240,52,1040,56   新建数据库 · prod-mysql · 已测试 18ms
操作栏 240,108,1040,44  校验 | 重置 | 创建
页签   240,152,1040,36  常规 | 编码 | 选项 | 变更计划
内容   252,200,688,344  两列表单，标签 160px，字段 320px
帮助   952,200,316,344  驱动说明、默认值、风险提示
结果   252,602,1016,168 校验 / 执行结果
```

表单坐标：

| 字段 | x | y | w | h |
|---|---:|---:|---:|---:|
| 数据库名 | 412 | 204 | 320 | 30 |
| 注释 | 412 | 242 | 460 | 64 |
| 字符集 / 编码 | 412 | 334 | 240 | 30 |
| 排序规则 / 区域 | 412 | 372 | 240 | 30 |
| 表空间 / 所有者 | 412 | 464 | 240 | 30 |

步骤：

1. 用户从连接节点进入新建数据库。
2. 表单自动带入当前连接和驱动。
3. 必填字段只突出数据库名；高级字段按驱动显示。
4. 用户输入名称后即时校验命名规则和必填项。
5. `变更计划` 展示目标、能力、风险和执行顺序，不切换到查询编辑器。
6. `Create` 执行后刷新对象树。
7. 成功后选中新数据库，并在右侧打开数据库详情。

驱动差异：

| 驱动 | 关键字段 | 默认策略 |
|---|---|---|
| MySQL/MariaDB/TiDB | database name、charset、collation | `utf8mb4` + `utf8mb4_bin` |
| PostgreSQL | database name、owner、template、encoding、locale、tablespace | 继承连接默认项 |
| SQL Server | database name、collation、owner、file options | 简化版先只暴露 name/collation |
| MongoDB | database name、initial collection | 数据库在首个 collection 创建时落地 |

## 流程二：修改数据库

入口：

- 数据库节点右键 -> `设计数据库`
- 数据库详情页动作条 -> `Modify`
- 数据库节点右键 -> 快捷动作：`Rename`、`Change Charset`

像素级布局：

```text
标题栏 240,52,1040,56   数据库设计器 · app_reporting · 已加载 42ms
操作栏 240,108,1040,44  保存草稿 | 校验 | 回退 | 应用变更 | 更多
页签   240,152,1040,36  常规 | 编码 | 对象 | 权限 | 变更计划 | 历史
内容   252,200,620,344  属性表单
检查器 884,200,384,344  当前值 / 新值 / 驱动支持说明
计划   252,602,1016,168 变更计划 + 风险 + 结果
```

步骤：

1. 用户选中数据库并打开设计器。
2. 系统加载实时数据库元数据，而不是只读缓存。
3. 用户修改名称、字符集、排序规则、注释或可支持属性。
4. 每个修改进入变更计划。
5. 应用变更前展示变更摘要。
6. 对高风险变更追加确认，例如数据库重命名、字符集变更。
7. 执行后刷新对象树和右侧详情。

规则：

- MySQL 类数据库 rename 不能伪装成单条通用 rename；必须显示迁移计划或不支持原因。
- PostgreSQL/SQL Server 可支持原生命名变更时走驱动策略。
- MongoDB database rename 没有通用原子语义时，不提供假按钮；提供“复制到新库/迁移计划”作为后续能力。

## 流程三：新建数据表

入口：

- 数据库或 schema 节点右键 -> `新建数据表`
- Tables group 右键 -> `New Table`
- 顶栏 `New` -> `Table`

像素级布局：

```text
标题栏     240,52,1040,56   表设计器 · 新建表 · 目标 public
操作栏     240,108,1040,44  保存草稿 | 校验 | 回退 | 创建表 | 更多
名称行     252,160,1016,40  名称 360px + 注释 520px
页签       240,200,1040,36  字段 | 索引 | 约束 | 选项 | 变更计划
表格       252,248,1016,298 字段表格
表格工具   252,546,1016,36  新增字段 | 删除字段 | 复制 | 上移 | 下移
计划       252,602,1016,168 校验 + 变更摘要 + 风险 + 结果
```

字段表格列宽：

| 列 | 宽度 |
|---|---:|
| 状态/PK | 48 |
| 名称 | 160 |
| 类型 | 170 |
| 长度 | 70 |
| 精度 | 80 |
| Scale | 64 |
| Not Null | 76 |
| Auto Inc | 76 |
| Default | 150 |
| Comment | 160 |
| Action | 110 |

超出宽度时横向滚动，列宽不自动压缩到不可读。

步骤：

1. 用户从数据库/schema/Tables 节点进入。
2. 系统带入目标 database/schema，不要求用户重复选择。
3. 默认创建一个 `id` 字段草稿，可由用户删除。
4. 用户在 Fields tab 中编辑字段网格。
5. Indexes tab 选择字段、索引类型、唯一性、排序。
6. Constraints tab 配置主键、外键、唯一约束、检查约束。
7. Options tab 配置 engine、charset、collation、tablespace、comment 等驱动字段。
8. Change Plan tab 展示对象、字段、索引、约束、驱动能力和风险。
9. Create Table 执行后刷新对象树并选中新表。

字段网格交互：

- Enter 提交当前单元格并移动到下一列。
- Tab 横向移动。
- Ctrl+S 保存。
- 右键字段行显示 duplicate、delete、set primary key、move。
- 类型列必须是驱动感知下拉，不能只靠自由文本。

## 流程四：修改数据表

入口：

- 表节点右键 -> `设计数据表`
- 表详情页 -> `Design`
- 数据浏览页顶部 -> `Structure`

像素级布局：

```text
标题栏     240,52,1040,56   表设计器 · public.orders · 行数 124,810
操作栏     240,108,1040,44  保存草稿 | 校验 | 回退选中项 | 回退全部 | 应用变更
页签       240,152,1040,36  字段 | 索引 | 约束 | 选项 | 变更计划 | 历史
表格       252,200,1016,344 可编辑结构表格
属性栏     252,552,1016,42  当前选中字段/索引的行内属性
计划       252,602,1016,168 变更计划 + 风险 + 执行结果
```

步骤：

1. 用户打开表设计器。
2. 系统加载实时表结构、索引、约束和表选项。
3. 用户编辑字段、索引、约束或表选项。
4. 每个改动保留原值、当前值和动作类型。
5. `Change Plan` 按执行顺序展示。
6. 高风险变更打标：删除字段、改类型、改 nullable、删除索引、删除约束、移动字段。
7. `Apply Changes` 前必须确认高风险项。
8. 执行成功后刷新表结构和对象树，保留设计器打开。

变更状态标记：

| 标记 | 含义 |
|---|---|
| `=` | 未变化 |
| `+` | 新增 |
| `~` | 修改 |
| `-` | 删除，必须确认 |
| `!` | 高风险或驱动不完全支持 |

## 统一动作条

所有 schema 设计器都使用同一动作条：

```text
[Save Draft] [Validate] [Revert] [Apply Changes] [More]
```

语义：

- `Save Draft`：保存未执行草稿，后续再做。
- `Validate`：检查命名、类型、能力、风险和阻塞项。
- `Revert`：回退全部或选中变更。
- `Apply Changes`：校验、确认、执行。
- `More`：查看历史、打开能力说明、导出结构报告。

第一阶段可以不做 `Save Draft` 持久化，但 UI 语义要预留，避免后续改交互。

动作条按钮位置：

| 按钮 | x 规则 | w | h |
|---|---|---:|---:|
| Save Draft | 左侧 252 | 96 | 30 |
| Validate | Save Draft 后 8px | 96 | 30 |
| Revert / Revert Selected | Validate 后 8px | 120 | 30 |
| More | Revert 后 8px | 72 | 30 |
| Apply/Create 主按钮 | 右侧 1268 - w | 132 | 30 |

主按钮永远在动作条最右侧；危险执行按钮使用红色 outline + 确认弹窗，不直接使用实心红。

## 执行反馈

执行完成后显示统一结果：

```text
Operation Result
Status: Success
Executed: 2 structure actions
Elapsed: 38ms
Metadata Refresh: 71ms
Target: prod-pg / app_db / public / orders

[View Log] [Refresh Metadata] [Open Object]
```

失败时必须显示：

- 失败阶段：validation、preview、execute、refresh。
- 数据库错误码或错误类。
- 可读错误消息。
- 是否已部分执行。
- 下一步建议：回退草稿、刷新元数据、重试。

## 与现有设计的关系

- 承接 `unix-workbench-ux` 的对象模型。
- 细化 `object-workbench` 中 create database/create table/alter table 的使用方式。
- 不改变 `query-console` 的定位：SQL 编辑器不是所有 schema 操作的主入口。
- 与 Navicat 对齐的是操作节奏、对象浏览器入口、结构化设计器、tab 分区、变更计划和执行反馈；Akt 保持自己的品牌、配色和图标资产。
